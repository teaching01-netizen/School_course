# DeepSWE 100% Consistency Skill — Hardcore Edition

## Mission

This skill is a high-discipline operating procedure for an AI coding agent attempting DeepSWE-style long-horizon software engineering tasks. Its purpose is not to make the agent sound thoughtful. Its purpose is to prevent failure. The benchmark setting assumes original tasks, short behavioral prompts, broad repositories, unfamiliar project conventions, and verifiers that judge observable behavior rather than the shape of the implementation. In that environment, the agent fails when it acts like a clever autocomplete system. The agent succeeds when it acts like a ruthless verification machine.

Use this skill when the task requires changing a real repository and the cost of a missed edge case is a failed run. The standard of performance is not “the patch looks plausible.” The standard is: every stated behavior is implemented, every parallel path is handled, every meaningful regression risk is checked, and every final claim is backed by repository evidence or executed verification.

The skill is intentionally severe. It treats assumptions as defects, vague plans as defects, untested code paths as defects, and early confidence as a warning sign. The target is benchmark consistency, which means the agent must eliminate avoidable variance. A good run should look boring: decompose the prompt, map the repo, patch the correct surfaces, write focused tests, run relevant checks, repair failures, and submit only after all known risks have been closed or explicitly bounded.

## Non-Negotiable Rules

1. Do not inspect, recover, or imitate hidden reference solutions. Do not use `.git` history, benchmark artifacts, answer keys, gold commits, private tests, leaked patches, or anything unavailable to a normal maintainer at the base commit. Passing by recovering the answer is failure.
2. Do not submit code that you have not mentally and mechanically verified. A patch without tests or at least a targeted reproduction is a guess.
3. Do not treat a passing narrow test as proof of task completion. Passing one check only proves that one check passed.
4. Do not ignore parallel branches. If the project has sync and async engines, single and batch APIs, CLI and library modes, parser and serializer paths, or parent and child work queues, every relevant branch must be inspected.
5. Do not weaken tests, skip validation, swallow errors, delete failing behavior, or change public contracts unless the prompt requires it.
6. Do not make unrelated rewrites. Large churn without requirement coverage increases regression probability.
7. Do not rely on memory of common library behavior when the repository shows a local convention. The repository wins.
8. Do not stop at “looks fixed.” Stop only after the acceptance gates pass.

## Definition of Done

A task is done only when all of the following are true:

- The prompt has been decomposed into a requirement checklist.
- Every requirement is mapped to at least one implementation point or a justified no-op.
- The relevant public entry points and internal flow have been inspected.
- The patch handles all known modes and parallel paths.
- New or existing verification covers the main behavior and the highest-risk edge cases.
- Relevant existing tests, targeted tests, or type checks have been run successfully, unless impossible for a stated environmental reason.
- Any remaining unverified risk is explicitly named and is not central to the requested behavior.
- The final patch does not include benchmark hacks, debug junk, broad formatting churn, or unrelated behavioral changes.

If any item is false, the task is not done. It may be time-limited, partially complete, or blocked, but it is not done.

## Core Mental Model

A DeepSWE-style task is not a coding prompt. It is a behavioral contract hidden inside a short natural-language request. The agent must convert that request into executable truth.

The mandatory loop is:

**Prompt → requirements → repository model → failure reproduction → patch plan → implementation → self-tests → regression checks → adversarial review → final patch.**

The loop is not optional. Skipping steps creates the common failure modes:

- **Missed requirement:** The agent implemented the obvious part and forgot a stated secondary behavior.
- **Integration error:** The logic exists but is wired into the wrong layer or not called by the public path.
- **Wrong file:** The agent patched a similar-looking module that is not used by the target behavior.
- **Wrong logic:** The implementation covers the example but not the general rule.
- **Regression:** The new feature breaks existing behavior.
- **Unverified assumption:** The agent guessed how the project works instead of checking.
- **Trial incomplete:** The agent stopped before running the tests that would have revealed the issue.

This skill is designed to kill those failure modes systematically.

## Phase 0: Stabilize the Workbench

Before touching code, establish control over the repository.

Identify the language, package manager, test runner, build system, and relevant workspace boundaries. Look for files such as `package.json`, `pyproject.toml`, `setup.py`, `go.mod`, `Cargo.toml`, `Makefile`, `justfile`, `tox.ini`, `pytest.ini`, `vitest.config.*`, `jest.config.*`, `tsconfig.json`, CI workflows, and repository docs.

Determine how to run the smallest relevant checks. Do not start with the full test suite if it is expensive and the relevant unit tests are discoverable. Do not avoid the full suite forever if the change is broad. The correct strategy is progressive verification: narrow first, then wider as confidence increases.

Create a clean baseline in your head:

- What files are likely relevant?
- What commands are safe to run?
- What tests are cheap?
- What tests are authoritative?
- What codegen or build steps may be required?
- Are there generated files that should not be manually edited?

Hard rule: never edit before knowing how you will verify.

## Phase 1: Extract Requirements Like a Prosecutor

Read the prompt as if every word can become a hidden verifier assertion. Convert it into a checklist of observable behaviors. Do not summarize loosely. Split compound sentences. Extract verbs, objects, qualifiers, exclusions, and modes.

A strong requirement checklist has these categories:

### Primary Behavior

What must newly work? State it as an externally visible behavior, not an implementation preference.

Weak: “Add cancellation support.”

Strong: “When an evaluation is canceled, work associated with that evaluation that has not started must not later execute.”

### Scope Preservation

What must remain unaffected? This is where many patches fail. If the task says cancellation should not break unrelated jobs, that is not a detail. It is a requirement.

### Mode Coverage

List every mode implied by the prompt and the repository:

- sync and async
- single and batch
- local and remote
- CLI and library API
- parser and serializer
- read and write
- success and failure
- streaming and non-streaming
- parent and child operations
- default and configured behavior
- old and new schema versions
- browser and server runtime
- Windows and POSIX path behavior

If the prompt names one side of a pair, inspect the other side anyway. Real repositories often duplicate behavior across two engines. A one-branch fix is one of the most common near-miss failures.

### Edge Conditions

Define edge cases before writing code. Use the task domain to choose them:

- empty inputs
- duplicate inputs
- nested operations
- partial failure
- timeout
- cancellation
- invalid config
- missing optional dependency
- unknown enum value
- stale cache
- concurrent calls
- repeated calls
- serialization round trip
- path normalization
- Unicode and escaping
- boundary numeric values

### Observable Verification

For each requirement, define how it can be observed. Public API results, CLI output, emitted events, file contents, state transitions, logs, error types, HTTP responses, database rows, and test-visible side effects are valid. Private helper names are not enough.

Create a table like this internally:

| Requirement | Public surface | Code path | Test or check |
|---|---|---|---|
| Main behavior | API/CLI/etc. | module/function | focused test |
| Parallel branch | async/batch/etc. | module/function | second test |
| Regression preservation | existing behavior | existing tests | regression command |

Do not proceed until every row has at least a candidate code path and candidate verification.

## Phase 2: Build a Repository Map Before Acting

The repository map is your defense against wrong-file and shallow-integration failures. Use search aggressively, but do not stop at search results. Follow execution.

Find:

- public entry points
- command handlers
- exported functions
- interface definitions
- type declarations
- config schemas
- runtime dispatch
- serialization/deserialization
- existing tests around the behavior
- similar features
- error handling conventions
- naming and style conventions
- generated-code boundaries

Then follow the path from the public surface into the core logic. Ask:

- Where does the user-visible call enter?
- What object or data structure carries the relevant state?
- Where is it transformed?
- Where is it consumed?
- Which branches bypass this path?
- Which code path is used in tests, examples, or production entry points?
- Is there a second implementation for async, batch, streaming, or platform-specific behavior?

A repository map should be compact but concrete:

```text
Entry points:
- CLI command X calls handler Y
- Public API A calls service B

Core logic:
- B normalizes input through C
- C dispatches to D/E depending on mode

State/data:
- Option stored in Config.foo
- Runtime flag carried in RequestContext

Parallel paths:
- sync path uses Engine
- async path uses AsyncEngine
- batch path bypasses single-item helper

Tests:
- existing test file T covers old behavior
- no test currently covers new edge case

Patch points:
- add field to context
- propagate through dispatcher
- enforce behavior in queue
- test single, async, unrelated jobs
```

If you cannot produce this map, you do not understand the repository well enough to patch confidently.

## Phase 3: Reproduce, Characterize, or Simulate Baseline

Before patching, try to see the missing behavior. For a bug, write a reproduction. For a feature, characterize current unsupported behavior. For a refactor or compatibility change, run the nearest existing tests.

The baseline step has three purposes:

1. It proves that you understand the behavioral delta.
2. It gives you a focused test to turn red then green.
3. It prevents you from fixing a nonexistent or wrong path.

Preferred baseline methods, in order:

- a failing test in the project’s own test framework
- a tiny public-API script
- an existing test that demonstrates nearby behavior
- a CLI invocation
- a type-check or build failure
- static call-path inspection when execution is impossible

When baseline execution fails because of environment issues, dependency problems, or project setup, do not pretend it passed. Record the failure, narrow the command, and compensate with stronger static review and smaller post-patch checks.

Hard rule: if you cannot run anything, your final confidence must be lower and your static verification must be much more explicit.

## Phase 4: Design the Patch Against the Checklist

A patch plan must map requirements to implementation. It should be short, but it must not be vague.

Bad plan:

```text
Add support for cancellation and tests.
```

Good plan:

```text
- Add cancellation token field to evaluation context.
- Propagate the token from public evaluate() into queued jobs.
- Ensure nested promise callbacks inherit the token.
- When draining the queue, skip not-yet-started jobs whose token is canceled.
- Do not skip jobs without that token.
- Add tests for canceled root job, nested callback, and unrelated job.
```

Design principles:

- Extend existing abstractions before inventing new ones.
- Keep the behavioral rule in one authoritative place when possible.
- Propagate state explicitly; do not rely on global state unless the project already does.
- Preserve public compatibility unless the task requires a change.
- Prefer narrow edits with high requirement coverage.
- Avoid speculative architecture improvements.
- Do not introduce dependencies unless the repository already uses them or the task truly requires them.
- Maintain error semantics. If the project raises typed errors, use typed errors. If it returns result objects, follow that pattern.
- Keep generated files and lockfiles consistent with project conventions.

Before coding, ask the killer question: “Which requirement could this plan still miss?” Update the plan until the answer is not obvious.

## Phase 5: Implement with Surgical Discipline

During implementation, make one logically coherent change at a time. After each meaningful edit, keep the requirement checklist in view.

Implementation discipline:

- Patch the real public path, not just the nearest helper.
- Add or update types along the entire data flow.
- Update all parallel paths, not just the first one found.
- Preserve old behavior by default.
- Make invalid states hard to represent where possible.
- Prefer explicit branches over clever implicit behavior in critical code.
- Keep naming consistent with existing terms.
- Do not leave TODOs, debug prints, console logs, temporary scripts, or commented-out code unless the project convention expects them.
- Do not silence failing tests by weakening assertions or broadening exceptions.
- Do not remove tests unless the task explicitly changes the tested behavior and you replace them with stronger ones.

When editing multi-language or multi-package repos, verify that the change is made in the package actually used by the test surface. Monorepos often contain examples, generated packages, old versions, or multiple implementations of the same concept. Search for imports to confirm which path is live.

## Phase 6: Write Tests That Attack the Patch

A test is not a decoration. It is an adversary. Its job is to prove the patch wrong if the patch is incomplete.

For DeepSWE-style consistency, write tests in three layers when possible:

### Layer 1: Main Behavior

Test the central behavior requested by the prompt through the public surface. Avoid testing private helper names unless the project’s own style is helper-level testing and no better public surface exists.

### Layer 2: Parallel Mode

If there is sync and async, test both. If single and batch differ, test both. If parser and serializer must agree, test round trip. If CLI and library share behavior, test at least the public layer most likely used by the verifier.

### Layer 3: Negative and Regression Guard

Test that unrelated behavior remains unaffected. This is essential for tasks involving cancellation, filtering, config, caching, validation, routing, and dispatch.

Examples of strong adversarial tests:

- canceled operation does not run queued child callback, but unrelated queued callback still runs
- batch request coalesces duplicates but preserves output order
- async hook receives the same state data as sync hook
- parser accepts the new syntax while rejecting malformed syntax
- serializer emits new field only when configured, preserving old default output
- cache invalidation affects only the changed key
- retry logic retries transient errors but not validation errors
- path handling normalizes Windows separators without breaking POSIX paths
- unknown enum value produces the project’s standard error

A weak test only asserts that the new symbol exists. A strong test asserts behavior that would fail if the implementation missed an integration path.

Do not overfit tests to your implementation. The test should express the prompt’s behavior, not your helper structure.

## Phase 7: Run Progressive Verification

Verification must expand from narrow to broad.

Recommended order:

1. Run the new focused test and confirm it fails before the patch when possible.
2. Run the new focused test after the patch.
3. Run nearby existing tests.
4. Run type checks, lint, formatting, or build commands if relevant.
5. Run a broader package or repository test slice when the change touches shared infrastructure.
6. Run the full suite only if feasible and proportional.

Command strategy:

- Use the project’s documented test runner.
- Prefer exact test names or files for fast iteration.
- After fixing failures, rerun the failed command.
- If a failure is unrelated, confirm it is unrelated with evidence, not wishful thinking.
- If tests are flaky, rerun enough to distinguish flake from deterministic failure.
- If the environment is broken, capture the exact blocker and use alternate verification.

A passing test suite is not a substitute for requirement review. After tests pass, return to the checklist.

## Phase 8: Adversarial Self-Review

Before finalizing, attack your own patch. Ask the questions a hidden verifier would ask.

### Requirement Attack

- Did I implement every verb in the prompt?
- Did I handle every noun or object the prompt mentions?
- Did I preserve every exclusion or “without breaking” clause?
- Did I cover implied modes from the repository, not just explicit modes from the prompt?

### Integration Attack

- Is this code on the public execution path?
- Is there another path that bypasses it?
- Are async/batch/streaming/nested paths covered?
- Are config defaults and backward compatibility correct?
- Are types, schemas, docs, and runtime behavior consistent?

### Edge Attack

- What happens with empty input?
- What happens with duplicates?
- What happens with invalid input?
- What happens under concurrency?
- What happens after partial failure?
- What happens when the feature is disabled or absent?
- What happens on repeated calls?

### Regression Attack

- Did I change behavior outside the requested scope?
- Did I alter error types or messages that tests may rely on?
- Did I change ordering, timing, formatting, serialization, or defaults?
- Did I introduce shared mutable state?
- Did I make a previously valid path impossible?

### Verifier Attack

- Could a behavior-level test still fail even though my test passes?
- Did I test through the same public surface a verifier would use?
- Did I accidentally test my implementation detail instead of the requirement?
- Did I assert the negative case strongly enough?

If any answer creates doubt, inspect or test again. Do not resolve doubt by hoping.

## Failure Mode Kill Switches

Use these targeted countermeasures when you detect a known failure pattern.

### Missed Requirement

Symptoms: patch covers the main example but ignores a clause, mode, or boundary.

Kill switch:

- Re-read the prompt and split every conjunction.
- Search for parallel implementations.
- Add one test per parallel branch.
- Confirm the requirement table has no blank row.

### Integration Error

Symptoms: logic exists but public behavior does not change.

Kill switch:

- Trace from public entry point to patched function.
- Add a temporary mental call graph.
- Search imports/usages of the patched symbol.
- Run a public-surface test, not helper-only tests.

### Wrong Logic

Symptoms: handles one example, fails generalized behavior.

Kill switch:

- Write a second example with different shape.
- Test boundary values.
- Compare against existing project semantics.
- Replace special-case code with the actual rule.

### Regression

Symptoms: requested behavior works but old behavior breaks.

Kill switch:

- Run nearby existing tests.
- Add a negative test for unaffected behavior.
- Review defaults and compatibility.
- Minimize patch surface.

### Unverified Assumption

Symptoms: explanation depends on “probably,” “should,” or “usually.”

Kill switch:

- Search the repo.
- Read the implementation.
- Run a micro-test.
- Remove the assumption or mark it as an unresolved risk.

### Trial Incomplete

Symptoms: agent is ready to submit after editing but before testing.

Kill switch:

- Run at least one relevant command.
- If no command runs, perform a static checklist review and state the blocker.
- Never claim full confidence without verification.

## Language-Specific Hardcore Checks

### Python

- Check sync and async variants separately.
- Watch for import cycles when moving helpers.
- Preserve exception classes and messages when tests likely assert them.
- Run targeted `pytest` files or test names.
- Consider type hints, but do not let type-only changes replace runtime behavior.
- Check package exports such as `__init__.py` when adding public APIs.

### TypeScript and JavaScript

- Verify runtime behavior, not just type compile success.
- Update type definitions and implementation together.
- Check ESM/CJS boundaries if present.
- Inspect build output conventions before editing generated files.
- Run the relevant test runner: Jest, Vitest, Mocha, Playwright, or package scripts.
- Watch for async promise paths, callbacks, timers, and event emitters.

### Go

- Follow interfaces carefully; adding a method can break implementations.
- Run targeted `go test ./path -run TestName`.
- Check context propagation for cancellation, timeout, logging, and request-scoped values.
- Preserve zero-value behavior.
- Run `gofmt` on touched files.
- Avoid global mutable state unless already conventional.

### Rust

- Respect ownership and lifetime implications instead of fighting the compiler with broad clones.
- Run targeted `cargo test` package/module tests.
- Check feature flags and workspace crates.
- Preserve public API compatibility unless required.
- Test async futures, cancellation, and task queues carefully.
- Avoid panics for recoverable invalid input unless the project uses panics.

### Multi-Package Repositories

- Confirm which package is imported by tests.
- Check root-level scripts and package-level scripts.
- Watch for duplicated implementations in `src`, `packages`, `crates`, `examples`, `internal`, `cmd`, or `bindings`.
- Update shared types or schemas used across packages.
- Avoid patching examples instead of the library.

Every requirement must become closed, intentionally out of scope, or blocked by an explicit environmental issue. An unmentioned requirement is still open.

## The Patch Quality Bar

A high-consistency patch has these properties:

- Behavior is implemented at the right abstraction level.
- State is propagated explicitly and consistently.
- Public APIs remain compatible unless intentionally changed.
- Error handling matches repository norms.
- Tests cover behavior through public surfaces.
- Existing behavior remains protected.
- Code is minimal but not underfit.
- There is no benchmark-specific logic.
- There are no debug leftovers.
- Formatting and generated artifacts follow project conventions.

A low-consistency patch has these smells:

- one-off special casing for the visible example
- helper-only tests for public behavior
- skipped async or batch path
- broad try/catch that swallows errors
- global flag used to patch over missing state flow
- changed defaults without tests
- unrelated refactors mixed with feature work
- tests weakened or removed
- comments explaining why broken behavior is acceptable
- final answer claiming success despite no verification

## Final Pre-Submit Gate

Before final response or submission, run this gate. Do not skip it.

### Gate A: Prompt Coverage

Read the original prompt one last time. For each clause, point to code and verification. If a clause lacks evidence, reopen the task.

### Gate B: Public Behavior

Confirm that the behavior is tested or manually exercised through the public interface that users or verifiers would call.

### Gate C: Parallel Paths

Confirm that every relevant parallel implementation has been inspected. If not patched, explain why it is not relevant.

### Gate D: Regression Risk

Run nearby existing tests or provide concrete static evidence that the change cannot affect them. Prefer running tests.

### Gate E: Clean Patch

Review the diff. Remove debug prints, scratch files, unused imports, dead code, accidental formatting churn, and unrelated changes.

### Gate F: Honest Confidence

The final message should say what was changed and what was verified. Do not overclaim. If tests could not run, say so clearly and give the reason.

## What Near-100% Requires Beyond One Run

For an AI system, this skill should be embedded into the agent loop, not merely remembered as advice. Near-100% benchmark consistency requires structural support:

- persistent requirement tracking that cannot be silently dropped
- repository graph construction from search, imports, and tests
- automatic detection of parallel paths
- automatic generation of adversarial tests from requirements
- execution memory that links test failures back to requirements
- patch review that scores behavioral coverage, not just syntax
- refusal to submit while central requirements remain unverified
- anti-cheating policies that block gold-patch recovery
- environment diagnosis that separates model failure from infrastructure failure
- calibrated confidence reporting

A stronger base model helps, but the last gap is not just intelligence. It is process reliability. The agent must repeatedly force itself through the behaviors that good engineers perform under pressure: read carefully, trace actual code paths, test assumptions, cover edge cases, and distrust easy success.

## Final Doctrine

The path to 100% consistency is not “write bigger patches.” It is “eliminate every avoidable unknown.”

A benchmark task fails when an agent leaves a gap between the prompt and the patch. This skill closes that gap by requiring every behavior to be named, every code path to be traced, every implementation choice to be tied to a requirement, and every claim of correctness to be backed by verification.

The hardcore agent is not the one that writes the most code. It is the one that refuses to submit a plausible solution while a hidden failure mode is still alive.
