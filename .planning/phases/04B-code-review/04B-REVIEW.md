## Summary

Reviewed the new `AbsenceHardDelete` flow end-to-end: DB method, HTTP handler, and two frontend callers. Compared against the mandatory reference pattern (`handleAbsenceStatusUpdate`). Found one blocker — a missing error response that causes silent 200 OK on malformed JSON input.

## Critical Issues

### CR-01: DecodeJSON failure silently returns 200 OK (no error written to client)

**File:** `backend/internal/httpapi/absenceshttp/management_routes.go:1353-1355`
**Issue:** When `DecodeJSON` fails (malformed JSON, missing body, oversized body), the handler returns without writing any response. `DecodeJSON` (adapter.go:166-169) does NOT write its own error — it just calls `json.NewDecoder(r.Body).Decode(v)` and returns the error. The handler then returns, leaving `w` with a default 200 OK and empty body. Every other handler in this file explicitly writes the error:
```go
// handleAbsenceStatusUpdate (line 624-627):
if err := s.a.DecodeJSON(w, r, &body); err != nil {
    s.a.WriteErr(w, http.StatusBadRequest, "bad_json", "Invalid JSON")
    return
}

// handleAbsenceDelete (line 1353-1355) — BUG:
if err := s.a.DecodeJSON(w, r, &body); err != nil {
    return  // ← no WriteErr → client gets 200 OK with empty body
}
```
This breaks the error contract: client cannot distinguish success from bad input. A curl with `--data-raw 'not json'` to `DELETE /api/v1/absences/:id` returns HTTP 200.

**Fix:**
```go
if err := s.a.DecodeJSON(w, r, &body); err != nil {
    s.a.WriteErr(w, http.StatusBadRequest, "bad_json", "Invalid JSON")
    return
}
```
