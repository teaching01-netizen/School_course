# Phase: 04-absence-architecture — Architecture Improvement Plan

## Overview

Decompose the monolithic `Absences.tsx` (546 lines) and `AbsenceDetail.tsx` (416 lines) into focused, testable sub-components in `src/components/absences/`. Consolidate duplicate modal logic shared between `Absences.tsx` and `KanbanView.tsx`. Extract shared utilities and constants. Wire pages back as thin orchestrators.

---

## Dependency Graph

```
Wave 1 (no dependencies):
  ├── AbsenceUtils.ts           — shared helpers (submittedAgo, titleCase, date displays, constants)
  ├── StatusBadge.tsx            — status badge presentational component
  ├── BatchProgressBar.tsx       — pure display: progress bar + failure state
  ├── AbsenceSelectionBar.tsx    — pure display: selection count + action buttons
  ├── AdminNotesEditor.tsx       — pure display: textarea + save button
  └── StatusTimeline.tsx         — pure display: timeline entries

Wave 2 (depends on StatusBadge, AbsenceUtils, existing UI primitives):
  ├── AbsenceFilters.tsx         — filter bar (SearchInput, selects, export button, view toggle)
  ├── AbsenceTable.tsx           — table body + rows (uses StatusBadge, dateSummary, sitInLabel)
  ├── AbsenceCards.tsx           — mobile card view (uses dateSummary, sitInLabel)
  ├── AbsenceSummaryCard.tsx     — detail: summary grid (uses date helpers)
  ├── SitInPlanCard.tsx          — detail: sit-in plan section
  ├── AbsenceProfileCard.tsx     — detail: student header + action bar
  ├── CancelModal.tsx            — shared: cancel modal (Modal + Button + constants)
  ├── DeleteModal.tsx            — shared: delete confirm modal (Modal + Button)
  ├── CancelAbsenceModal.tsx     — detail: simple cancel modal (Modal + Button)
  └── OverrideSitInModal.tsx     — detail: tabbed override dialog (Modal + Button + selects)

Wave 3 (wire everything back):
  ├── Absences.tsx               — thin orchestrator using Wave 1+2 components
  ├── AbsenceDetail.tsx          — thin orchestrator using Wave 1+2 components
  └── KanbanView.tsx             — consolidated: use shared CancelModal + DeleteModal
```

### Task 3.1: Rewrite `Absences.tsx` — thin orchestrator

**File:** `src/pages/Absences.tsx`

After extraction, the page becomes an orchestrator that:

1. **Owns all state** (searchParams, page, selected, reviewing, cancelTargets, deleteTarget, batch state)
2. **Owns all effects** (load, search param sync)
3. **Owns all API call handlers** (markReviewed, cancelAbsences, deleteAbsence, exportCsv, exportSelected, markSelectedReviewed, retryFailed)
4. **Renders extracted components** with state passed as props

Structure:
```typescript
export default function Absences() {
  // --- All state ---
  const { addToast } = useToast();
  const navigate = useNavigate();
  const [searchParams, setSearchParams] = useSearchParams();
  // ... (all state that existed)

  // --- All handlers --- (keep as anonymous or top-level functions)
  // Load, status updates, batch ops, export, delete, cancel

  // --- Render ---
  return (
    <div className="w-full">
      {/* Board mode or Table mode */}
      {viewMode === "board" ? (
        <>
          <AbsenceFilters ... board-mode props />
          <KanbanView filters={filters} />
        </>
      ) : (
        <>
          <AbsenceFilters ... table-mode props />
          <AbsenceSelectionBar ... />
          <BatchProgressBar ... />
          <AbsenceTable ... />
          {/* Pagination section — stays inline, too tightly coupled to state */}
          <CancelModal ... />
          <DeleteModal ... />
        </>
      )}
    </div>
  );
}
```

**What stays inline** (by design — too tightly coupled to orchestration state):
- Pagination (prev/next buttons, page input) — references local state directly
- Subjects memo — depends on page data shape + needs memo hook
- The requestQuery memo + load effect
- All event handlers (they reference multiple state variables and API calls)
